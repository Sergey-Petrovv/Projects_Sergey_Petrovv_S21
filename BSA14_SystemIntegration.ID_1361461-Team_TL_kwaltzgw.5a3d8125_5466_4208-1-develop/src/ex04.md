# Exercise 04 — Описание сценариев интеграции

Для каждого потока задачи 2 разработайте сценарий интеграции в виде UC или диаграммы.

# Описание сценариев интеграции


## UC-DF01 — Заказы от поставщиков (в систему: поставщик → оператор → система)

**Цель:** создать заказ в Системе доставки на основании данных, полученных от поставщика “любым доступным способом”.  
**Акторы:**

* Оператор (инициатор)
* Система доставки (получатель/обработчик)

**Триггер:** оператор получил новый заказ от поставщика и начинает ввод в систему.

**Предусловия:**

* Оператор авторизован в Системе доставки.
* В системе загружены/существуют точки комплектации (DF-02). 

**Основной сценарий:**

1. Оператор открывает форму “Создать заказ”.
2. Оператор вводит/заполняет поля заказа.
3. Оператор выбирает точку комплектации из справочника.
4. Система доставки валидирует обязательные поля и форматы.
5. Система доставки присваивает заказу внутренний идентификатор/регистрирует заказ.
6. Система доставки сохраняет заказ и делает его доступным для дальнейшей обработки.

**Альтернативы / исключения:**

* A1: Не найдена точка комплектации → оператор выбирает другую / заводит вручную (если разрешено), иначе ошибка.
* A2: Ошибка валидации (нет адреса/типа оплаты) →  заказ не создаётся.
* A3: Оператор отменил ввод → заказ не создаётся.

**Постусловия:**

* Новый заказ создан в Системе доставки и может попасть в пул “свободных” заказов.

## UC-DF02 — Адреса точек комплектации (разовая загрузка в систему)

**Цель:** выполнить **единовременную загрузку** справочника точек комплектации в Систему доставки.   
**Акторы:**

* Администратор
* Система доставки (получатель)
* Партнёрская компания (поставщик файла/данных)

**Триггер:** старт проекта / получен файл/выгрузка от партнёра для первичной загрузки.

**Предусловия:**

* Администратор авторизован и имеет право “Импорт справочников”.
* Получен источник данных (файл/таблица) от партнёра. 

**Основной сценарий:**

1. Администратор открывает интерфейс загрузки данных.
2. Загружает файл/набор данных партнёра.
3. Система доставки валидирует структуру и обязательные поля.
4. Система доставки создаёт записи справочника точек комплектации.
5. Система доставки формирует отчёт о загрузке (успешно/ошибки по строкам).

**Альтернативы / исключения:**

* A1: Ошибка формата файла → импорт не выполнен.
* A2: Частичные ошибки в строках → импортирует корректные строки, по ошибочным формирует протокол.
* A3: Дубликаты точек → правило: объединение по ключу/пропуск/ошибка.

**Постусловия:**

* Справочник точек комплектации доступен для выбора в DF-01 и отображения курьерам.

## UC-DF03 — Свободные заказы для курьера (из системы → в мобильное приложение)

**Цель:** предоставить курьеру список доступных заказов для выбора/бронирования.   
**Акторы:**.   

* Мобильное приложение курьера (инициатор запроса)
* Система доставки (поставщик данных)

**Триггер:** курьер открывает список свободных заказов / выполняет обновление списка.

**Предусловия:**

* Курьер аутентифицирован.
* Курьер активен и имеет право брать заказы.

**Основной сценарий:**

1. Приложение запрашивает список свободных заказов (по курьеру/зоне/фильтрам).
2. Система доставки отбирает заказы в статусе “свободен” по правилам.
3. Система доставки отправляет список.
4. Приложение отображает список.

**Альтернативы / исключения:**

* A1: Нет доступных заказов → пустой список.
* A2: Курьер не авторизован/заблокирован → просмотри списка свободных заказов недоступен.

**Постусловия:**

* Курьер видит актуальные свободные заказы.

## UC-DF04 — Статус выполнения заказа курьером (в систему из мобильного приложения)

**Цель:** принять от курьера событие изменения статуса заказа (забрал/доставил/ошибка).   
**Акторы:**

* Мобильное приложение курьера (инициатор)
* Система доставки (получатель)

**Триггер:** курьер меняет статус заказа в приложении.

**Предусловия:**

* Курьер аутентифицирован.
* Заказ назначен курьеру.

**Основной сценарий:**

1. Приложение отправляет Статус заказа.
2. Система доставки проверяет право курьера на заказ.
3. Система доставки валидирует допустимость перехода статусов.
4. Система доставки сохраняет статус и пишет событие в журнал.
5. Система доставки возвращает подтверждение.

**Альтернативы / исключения:**

* A1: Недопустимый переход → переход их данного статуса недостуен.
* A2: Дубль события → не возможен.

**Постусловия:**

* Статус заказа обновлён и доступен диспетчеру/выгрузкам.

## UC-DF05 — Данные о курьерах (в систему из интерфейса администратора)

**Цель:** зарегистрировать курьера и назначить права доступа для работы с системой.     
**Акторы:**

* Администратор (инициатор)
* Система доставки (получатель)

**Триггер:** требуется добавить нового курьера или изменить данные/права.

**Предусловия:**

* Администратор аутентифицирован и имеет права управления курьерами.

**Основной сценарий:**

1. Администратор открывает раздел “Курьеры”.
2. Создаёт/редактирует карточку курьера.
3. Система доставки валидирует обязательные поля и уникальности (например, телефон/логин).
4. Система доставки сохраняет карточку и права.
5. Система доставки подтверждает успешное сохранение.

**Альтернативы / исключения:**

* A1: Дубликат телефона/логина → сохранение отклонено.
* A2: Недостаточно прав у администратора → запрет.

**Постусловия:**

* Курьер зарегистрирован, может входить в приложение и работать с заказами.

## UC-DF06 — Команды переназначения заказа от диспетчера (в систему из интерфейса диспетчера)

**Цель:** переназначить заказ другому курьеру для контроля исполнения.  
**Акторы:**

* Диспетчер (инициатор)
* Система доставки (получатель)

**Триггер:** диспетчер обнаружил проблему исполнения и выполняет “Переназначить”.

**Предусловия:**

* Диспетчер аутентифицирован и имеет право переназначения.
* Заказ ещё не завершён (не Доставлен) и переназначение разрешено правилами.

**Основной сценарий:**

1. Диспетчер открывает карточку заказа.
2. Выбирает нового курьера и указывает причину/комментарий.
3. Система доставки проверяет допустимость переназначения (статус заказа, доступность нового курьера).
4. Система доставки фиксирует событие переназначения в журнале.
5. Система доставки обновляет назначение заказа (актуальный Курьер).

**Альтернативы / исключения:**

* A1: Заказ уже завершён → недоступно переназначений.
* A2: Новый курьер недоступен/неактивен → курьер не найден.
* A3: Конкурентное изменение назначения → диспетчер обновляет данные и повторяет действие.

**Постусловия:**

* Заказ назначен новому курьеру; в системе отображается актуальный исполнитель.

## UC-DF07 — Данные о заказе для бухгалтерии (из системы → в бухгалтерию)

**Цель:** передать в бухгалтерию данные о поступившем заказе для расчётов с поставщиками.   
**Акторы:**

* Система доставки (инициатор)
* Сторонняя ИТ-система бухгалтерии (получатель)

**Триггер:** заказ создан в системе (после ввода оператором) или наступило время регламентной выгрузки.

**Предусловия:**

* Заказ создан и содержит данные, необходимые бухгалтерии.
* Канал интеграции с бухгалтерией доступен.

**Основной сценарий:**

1. Система доставки формирует сообщение по заказу.
2. Отправляет сообщение в бухгалтерию.
3. Бухгалтерия валидирует и регистрирует данные.
4. Возвращает ACK/результат приёма.

**Альтернативы / исключения:**

* A1: Бухгалтерия недоступна → ретраи/очередь на повтор.
* A2: Дубль по orderId → ошибка.

**Постусловия:**

* Бухгалтерия знает о заказе и может учитывать обязательства перед поставщиком.

## UC-DF08 — Данные о доставке для бухгалтерии (из системы → в бухгалтерию)

**Цель:** передать факт доставки (или неуспех) для расчёта оплаты курьерам.    
**Акторы:**.   

* Система доставки (инициатор)
* Сторонняя ИТ-система бухгалтерии (получатель)

**Триггер:** заказ получил финальный статус доставки (например, Доставлен, Отменен) по DF-04.

**Предусловия:**

* В системе зафиксирован итог доставки и исполнитель .
* Канал интеграции с бухгалтерией доступен.

**Основной сценарий:**

1. Система доставки формирует Оплату курьеру за доставку.
2. Отправляет сообщение в бухгалтерию.
3. Бухгалтерия регистрирует факт доставки как основание для начисления.
4. Возвращает ACK.

**Альтернативы / исключения:**

* A1: Бухгалтерия недоступна → ретраи/очередь.
* A2: Дубль сообщения → идемпотентность по (orderId + deliveryStatus + deliveryDate) или messageId.

**Постусловия:**

* В бухгалтерии есть данные для расчёта оплаты курьеру.

## UC-DF09 — Данные о начисленной курьеру оплате (в систему из бухгалтерии)

**Цель:** принять начисления и отобразить их курьеру в личном кабинете/приложении.  
**Акторы:**

* Сторонняя ИТ-система бухгалтерии (инициатор)
* Система доставки (получатель)

**Триггер:** бухгалтерия выполнила расчёт начислений за период или отметила выплату.

**Предусловия:**

* Курьер зарегистрирован в Системе доставки (DF-05).
* Период задан в формате `YYYY-MM`.

**Основной сценарий:**

1. Бухгалтерия отправляет Оплату курьеру.
2. Система доставки валидирует данные.
3. Система доставки сохраняет начисление.
4. Система доставки подтверждает приём.

**Альтернативы / исключения:**

* A1: courierId не найден → ошибка.
* A2: Дубль сообщения → ошибка.

**Постусловия:**

* Начисления/выплаты доступны курьеру для просмотра.
